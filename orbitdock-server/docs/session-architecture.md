# Session Architecture

How sessions flow through OrbitDock — from creation to message persistence to restoration after crashes.

## Session Modes

Every session has a **provider** (Claude or Codex) and an **integration mode** (direct or passive):

```
                 ┌─────────────┬──────────────────┐
                 │   Direct    │     Passive       │
├────────────────┼─────────────┼──────────────────┤
│ Claude         │ SDK stdin/  │ Hook HTTP POST    │
│                │ stdout NDJSON│ + transcript sync │
├────────────────┼─────────────┼──────────────────┤
│ Codex          │ codex-core  │ Rollout file      │
│                │ embedded    │ watcher (FSEvents) │
└────────────────┴─────────────┴──────────────────┘
```

**Direct** — OrbitDock owns the AI process. It spawns the CLI/runtime, pipes messages through it, and controls the full lifecycle. Messages flow through the connector's event loop.

**Passive** — The AI runs independently (user's terminal). OrbitDock observes via hooks (Claude) or filesystem watching (Codex). Messages are loaded from transcript/rollout files.

## Data Flow Overview

```
DIRECT SESSIONS:
  SwiftUI app
    ↓ WebSocket: CreateSession
  orbitdock-server
    ↓ spawns connector
  Claude CLI / codex-core
    ↓ stdout events / codex events
  ConnectorEvent
    ↓ transition(state, input) → effects
  Effect::Persist → PersistCommand → batch buffer → SQLite (500ms / 50 msgs)
  Effect::Emit   → broadcast channel → WebSocket → SwiftUI app

PASSIVE CLAUDE:
  User's terminal (claude CLI)
    ↓ hooks fire
  HTTP POST /api/hook
    ↓ hook_handler.rs
  Session metadata updated (work_status, last_tool, etc.)
    ↓ sync_transcript_messages()
  ~/.claude/projects/{hash}/{session}.jsonl
    ↓ parse JSONL → Message structs
  SessionCommand::AddMessageAndBroadcast
    ↓
  broadcast → WebSocket → SwiftUI app

PASSIVE CODEX:
  User's terminal (codex CLI)
    ↓ writes to rollout file
  ~/.codex/sessions/rollout-{id}.jsonl
    ↓ FSEvents watcher (rollout_watcher.rs)
  Parse JSONL events → Message structs
    ↓
  SessionCommand::AddMessageAndBroadcast + PersistCommand::MessageAppend
    ↓
  broadcast → WebSocket → SwiftUI app
  batch buffer → SQLite
```

## Direct Claude Sessions

### Creation

1. Swift app sends `CreateSession { provider: Claude, cwd, model }` over WebSocket
2. Server generates OrbitDock session ID, creates `SessionHandle` — `websocket.rs:1086`
3. `PersistCommand::SessionCreate` sent to batch writer (async, not immediate)
4. Spawns `ClaudeConnector` which starts the `claude` CLI subprocess — `claude.rs:167`

### Claude CLI Subprocess

The connector spawns the CLI with streaming JSON protocol:

```bash
claude --output-format stream-json --verbose --input-format stream-json \
  --permission-prompt-tool stdio --model <model> --permission-mode <mode>
```

Three async tasks start immediately — `claude.rs:264-341`:
1. **stderr reader + exit watcher** — captures stderr, detects process death
2. **stdin writer** — receives messages from the session actor, writes to CLI stdin
3. **stdout reader (event_loop)** — reads NDJSON from CLI stdout, produces `ConnectorEvent`s

### Initialization Handshake

After spawning, the connector sends an `Initialize` control request — `claude.rs:580`:

```
OrbitDock → stdin:  {"type":"control","command":{"type":"Initialize",...}}
CLI → stdout:       {"type":"system","subtype":"hook_started","session_id":"<hook-id>"}
CLI → stdout:       {"type":"control_response","response":{"subtype":"success",...}}
CLI → stdout:       {"type":"system","subtype":"init","session_id":"<sdk-session-id>",...}
```

The `init` event is **critical** — it provides:
- `session_id` — the Claude SDK session ID (stored as `claude_sdk_session_id`)
- Slash commands and skills discovery
- Confirmation that the CLI is ready for prompts

**30-second timeout** on the Initialize response — `claude.rs:600`. If the CLI doesn't respond, connector creation fails.

### Message Flow

```
CLI stdout → event_loop reads line → dispatch_stdout_message()
  → ConnectorEvent::MessageCreated / MessageUpdated / TurnStarted / etc.
  → mpsc channel (256 buffer) → ClaudeSession::start_event_loop()
  → handle_event_direct() → transition(state, input)
  → Effect::Persist(MessageAppend) → persist_tx → batch writer → SQLite
  → Effect::Emit(MessageAppended)  → broadcast   → WebSocket → Swift app
```

Key code paths:
- stdout reading (event_loop): `claude.rs:665`
- Event dispatch (dispatch_stdout_message): `claude.rs:779`
- Session event loop: `claude_session.rs:168` (start_event_loop)
- Transition: `claude_session.rs:347` (handle_event_direct), call at `358`
- Effect application: `claude_session.rs:375`

### Session ID Mapping

A direct Claude session has **three** IDs:

| ID | Source | Example |
|----|--------|---------|
| OrbitDock session ID | Generated by server | `ed7449c6-4f99-...` |
| Claude SDK session ID | From `claude.init` event | `6831cfb3-8572-...` |
| Hook session ID | From `hook_started` event | `9de9ecda-155b-...` |

The hook session ID gets registered as a "managed thread" so that if hooks arrive via HTTP POST for this same CLI session, they're routed to the existing direct session instead of creating a duplicate passive session — `claude_session.rs:201`.

### Resume Flow

When resuming an ended direct Claude session:
1. Swift app sends `ResumeSession { session_id }` over WebSocket
2. Server spawns new `ClaudeConnector` with `--resume <claude_sdk_session_id>` — `claude.rs:191`
3. CLI restores its conversation context and emits `init` again
4. Event loop restarts, new messages flow through the same pipeline

**Requires `claude_sdk_session_id` to be persisted.** If the CLI never emitted `init` (stalled initialization), this column is NULL and resume is impossible.

## Direct Codex Sessions

### Creation

1. Swift app sends `CreateSession { provider: Codex, cwd, model }` over WebSocket
2. Server generates session ID, creates `SessionHandle` — `websocket.rs:1086`
3. Spawns `CodexConnector` which creates a codex-core thread — `codex.rs:68`:
   - Creates `ThreadManager` with `SessionSource::Mcp`
   - Calls `thread_manager.start_thread(config)` → creates rollout file at `~/.codex/sessions/`
   - Gets back `CodexThread` handle

### Event Flow

```
codex-core runtime → CodexThread::next_event()
  → translate to ConnectorEvent (codex.rs:328-1237)
  → mpsc channel → CodexSession::start_event_loop()
  → handle_event_direct() → transition(state, input)
  → Effect::Persist / Effect::Emit (same as Claude)
```

### Codex vs Claude Differences

| Aspect | Claude Direct | Codex Direct |
|--------|--------------|--------------|
| Process | Spawns CLI subprocess | Embeds codex-core library |
| Communication | stdin/stdout NDJSON | In-process Rust API |
| Rollout file | None (transcript is CLI-side) | Created by codex-core |
| Message IDs | Stable per session (from SDK) | Generated fresh on create/resume |
| Resume | `--resume <sdk_session_id>` | `resume_thread_from_rollout()` |
| Steering | Not supported | `steer_turn()` injects guidance mid-turn |

### Resume

On resume, `CodexConnector::resume()` — `codex.rs:107-160`:
1. Finds rollout file via `codex_core::find_thread_path_by_id_str()`
2. Calls `thread_manager.resume_thread_from_rollout(config, rollout_path)`
3. codex-core replays history from rollout file
4. Event loop restarts

**Note:** Codex generates **new message IDs** on resume, potentially creating duplicates in the DB. Claude's IDs are stable.

## Passive Claude Sessions (Hook-Based)

### Session Creation

Claude hooks arrive via HTTP POST to `/api/hook` — `hook_handler.rs`:

1. `SessionStart` hook → **deferred** in `PendingClaudeSession` cache (not materialized yet) — `hook_handler.rs:81`
2. First actionable hook (`StatusEvent` or `ToolEvent`) → `materialize_claude_session()` — `hook_handler.rs:1126`
3. Session created with `claude_integration_mode = Passive`

Deferred creation prevents ghost sessions from `claude -c` bootstrap processes that start but never do anything.

### Message Loading

Passive Claude sessions do **NOT** persist messages to SQLite. Instead:

1. Each hook event triggers `sync_transcript_messages()` — `websocket.rs:4236`
2. This reads the Claude transcript file at `~/.claude/projects/-{cwd_normalized}/{session_id}.jsonl`
3. Compares message count: if new messages exist, they're broadcast to subscribers
4. Messages are parsed from JSONL with tool_use/tool_result pairing — `persistence.rs:1932`

### Transcript Path

Set via `claude_transcript_path_from_cwd(cwd, session_id)`:
```
~/.claude/projects/-Users-bob-project/{session_id}.jsonl
```

This path is stored in the `transcript_path` column and used as the message source on restore.

## Passive Codex Sessions (Rollout Watcher)

### File Watching

`rollout_watcher.rs` monitors `~/.codex/sessions/` via FSEvents:

1. New `rollout-*.jsonl` detected → read `session_meta` header → create session
2. Subsequent lines → parse events → `append_chat_message()` — `rollout_watcher.rs:1016-1070`
3. Messages **ARE** persisted to SQLite via `PersistCommand::MessageAppend`

### Difference from Passive Claude

| Aspect | Passive Claude | Passive Codex |
|--------|---------------|---------------|
| Message source | Transcript file (re-read each time) | Rollout file (incremental tail) |
| DB persistence | Metadata only (no messages) | Messages persisted to DB |
| Restore | Load from transcript file | Load from DB |
| Deduplication | Message count comparison | Offset tracking in `codex-rollout-state.json` |

## Persistence Architecture

### Batch Writer

All DB writes go through `PersistenceWriter` — `persistence.rs:316`:

```rust
PersistenceWriter {
    batch: Vec<PersistCommand>,
    batch_size: 50,           // flush after 50 commands
    flush_interval: 100ms,    // OR flush after 100ms
}
```

Whichever threshold hits first triggers a flush. The flush opens a single SQLite transaction and executes all commands in order.

### The Data Loss Window

```
    Event arrives
        │
        ▼
    transition() produces Effect::Persist + Effect::Emit
        │                          │
        ▼                          ▼
    PersistCommand sent       Broadcast to clients
    to batch channel          (IMMEDIATE)
        │
        ▼
    Sits in batch buffer
    (up to 100ms / 50 cmds)
        │
        ▼
    flush_batch() → SQLite transaction
```

**If the server crashes between Effect::Emit and flush_batch():**
- Clients saw the message (it was broadcast)
- But it was never written to SQLite
- On restart, message is gone

This affects **direct sessions** most severely because they have no external transcript file to fall back on. Passive Claude sessions re-read the transcript; passive Codex sessions already wrote to DB incrementally.

## Server Restart & Session Restoration

### Startup Flow — `main.rs:265`

All session restoration happens in `load_sessions_for_startup()` — `persistence.rs:2295`:

1. Run migrations
2. **Cleanup stale sessions** (within `load_sessions_for_startup`):
   - End stale passive Codex sessions (idle >15min, not awaiting approval)
   - End empty Claude shells (0 prompts, 0 tools, no messages)
3. **Load sessions**:
   - Query: active sessions, OR ended by server_shutdown, OR active in last 7 days
   - For each session:
     - Load messages from `messages` table
     - If empty AND `transcript_path` exists → load from transcript file
     - Create `SessionHandle::restore()` with loaded messages
4. **Register sessions** in `SessionRegistry`

### Connector Restoration

Connectors are **NOT** restored at startup. They're created lazily when a client subscribes:

```
Client subscribes → is_direct_needing_connector?
  → Yes: spawn connector with --resume / resume_from_rollout
  → No:  just send snapshot
```

This means direct sessions appear in the sidebar immediately but the AI process only reconnects when you click on it.

### What Survives a Crash

| Session Type | Messages Survive? | How? |
|--------------|-------------------|------|
| Direct Claude | Partial | Only what was flushed to SQLite before crash |
| Direct Codex | Partial | Only what was flushed; rollout file has full history but isn't reconciled |
| Passive Claude | Yes | Transcript file is the source of truth, re-read on restore |
| Passive Codex | Yes | Messages are persisted to DB incrementally by watcher |

### What Doesn't Survive

1. **Unbatched messages** in the persistence writer's buffer
2. **Session metadata** that was only in-memory (e.g., `claude_sdk_session_id` if it arrived but wasn't flushed)
3. **Connector state** — all connectors are recreated on subscribe

## Known Issues

### Ghost Direct Sessions

If a direct Claude session's CLI stalls during initialization (never emits `claude.init`):
- Session exists in DB with `claude_integration_mode = direct`
- But `claude_sdk_session_id` is NULL, `prompt_count = 0`, `tool_count = 0`
- 0 messages in DB, no `transcript_path` set
- Shows "No messages yet" with no way to recover
- Resume is impossible (no SDK session ID)

**Root cause:** No initialization timeout beyond the 30s control request. If the CLI sends the control response but never sends `init`, the session sits idle forever.

**No guard against duplicate direct sessions.** Creating a second direct session for the same cwd while one is active can cause the CLI to stall on project-level locks.

### Message Loss on Server Rebuild

If you rebuild and restart the server while a direct session is active:
1. Server process dies → CLI subprocess dies (broken pipe)
2. Any messages in the 500ms batch buffer are lost
3. On restart, session is marked `ended` with `end_reason = cli_exited`
4. Messages loaded from DB — only what was flushed before crash
5. If ALL messages were in the buffer (unlikely for long sessions, possible for short ones), you get 0 messages

### No Transcript Fallback for Direct Claude

Direct Claude sessions don't set `transcript_path` because the Claude CLI manages its own transcript independently. On restore, if messages are missing from the DB, there's nowhere to recover them from.

The Claude CLI does write transcripts at `~/.claude/projects/{hash}/{sdk_session_id}.jsonl`, but the server doesn't know to look there for direct sessions because the path derivation depends on the hook session ID mapping.
